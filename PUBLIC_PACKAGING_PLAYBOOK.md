# Public Packaging Playbook

This is the public-facing packaging plan for `baishiqi45-dotcom`. It keeps the
headline on useful control methods and treats small repos as supporting slices.

## Positioning

**Short version**

> 白子烨是 AI 产品执行导演：把非工程师的商业意图、影像判断和创意素材，翻译成可被 AI coding agent、外部工程师、影像工具执行和验证的工作系统。

**English version**

> AI-native creative operator for intent-to-artifact systems: film/directing
> judgment, agent handoff, production control, artifact verification, and lesson
> writeback.

**Do not lead with**

- GitHub stars.
- "Open-source agent toolkit."
- "Epistemic OS."
- Sidecar orchestration.
- Provider, runtime, billing, or generation claims.
- "I can replace engineers."

**Lead with**

- AI video production control.
- Director-style agent workflows.
- Non-engineer AI product control.
- Evidence gates, retake logic, and owner acceptance boundaries.

## Website Structure

The personal website should be the main conversion surface. GitHub is the proof
appendix.

### Home

- One-line identity: AI 产品执行导演 / AI-native Creative Operator.
- One short paragraph: film/directing background plus AI-assisted production
  control.
- Three entry buttons:
  - AI video production control
  - Project Clinic for non-engineers
  - Workflow rescue and verification
- One featured case card above the fold.

### Work

Use 3-5 case studies. Each case should expose method, not private truth.

Case-card format:

```text
Title
Original intent
Where the project was losing control
Control method used
What was handed off
How the artifact was reviewed
What became reusable
Public boundary
```

### Methods

Publish small method cards:

- Five continuity locks for AI video.
- Failure-layer debug table for short drama.
- DirectorShotIR: performance, blocking, camera, transition, asset role, review.
- Project Clinic startup packet.
- Outcome-delta writeback note.
- Video admission ladder: source, static evidence, human playback, owner seal.

## GitHub Rules

- Profile README sells the person and method, not the repo list.
- Repos are proof appendices.
- Old tool repos should not ask for stars in their first screen.
- Public repos should contain synthetic demos, templates, and clear boundaries.
- Private project truth stays private.

## Publish Checklist

Before publishing a case or repo:

- [ ] No private customer name unless explicitly authorized.
- [ ] No raw script, prompt package, source media, subtitles, or screenshots from
      closed sources.
- [ ] No provider/account/job/credit/runtime state.
- [ ] No "final", "accepted", "validated", or "production-ready" language unless
      the owner acceptance evidence is public and current.
- [ ] Candidate, dry-run, fixture, and sidecar review are labeled as such.
- [ ] The public value is legible without internal system terminology.
